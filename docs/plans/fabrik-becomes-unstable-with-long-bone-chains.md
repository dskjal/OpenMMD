# IK を含む長いチェーンで不安定になるバグの原因

• 結論

  - 最有力の原因は、4a12025153d1802c70f7c385a9c1cf93c0eb4aaa で solveIk() の既定経路が CCD から FABRIK に切り替わっ
    たことです。対象の 髪ＩＫ_R/L は PMX 上で長い無制限チェーンなので、新しい FABRIK 経路にそのまま入っています。

  - 右髪２ / 左髪２ はそのチェーン内のリンクで、さらに剛体は type="2" の bone+physics なので、IK の揺れがそのまま物
    理へ伝わりやすい構造です。

  根拠

  - 変更コミットは source/ik.js のみが実質的なロジック変更で、物理エンジン側は触っていません。
  - source/ik.js:21 で solveIk() が _fabrikSolver() を呼ぶようになっています。
  - source/ik.js:93 以降の solveMultiLinkFabrikChain() は、位置を FABRIK で解いたあと applySolvedChainRotations() で
    各ボーンの回転を再構成します。

  - その再構成は source/ik.js:252 の applySolvedChainRotations() と source/ik.js:318 の
    buildRotationBetweenVectors() に依存し、dot <= -1 近傍では source/ik.js:356 の findOrthogonalAxis() が任意の直交
    軸を選びます。

  - ここが髪チェーンでは不安定化しやすいです。髪のような長い・ほぼ一直線・左右対称のチェーンでは、位置解は出ても「ね
    じれ」の自由度が残り、微小な数値差で回転軸が切り替わりやすいからです。これはコードからの推定です。

  - さらに source/ik.js:377 の refineChainWithCcd() は最大でも min(loopCount, 8) 回しか回らないため、髪ＩＫ_R/L のよ
    うな 7 リンク・無制限チェーンでは、旧 CCD より収束の余地が小さくなっています。

  - 実データでも、 髪ＩＫ_R/L は 7 リンクで全て hasLimit=0 でした。右髪２ / 左
    髪２ はそれぞれ boneIndex=126/136、剛体は index=27/35 で type="2"、ジョイントも直後にあります。

  - 参照箇所は source/ik.js:21、source/ik.js:93、source/ik.js:252、source/ik.js:318、source/ik.js:356、source/ik.js:377。

  要するに

  - 旧 CCD は「少しずつ回す」ので髪チェーンのねじれが比較的連続でした。
  - 新 FABRIK は「位置を解いてから回転を復元する」ため、髪のような無制限チェーンで回転の曖昧さが表に出やすく、右髪
    ２ / 左髪２ の剛体が不安定化した、というのが調査結果です。

  必要なら次に、修正方針を2案に絞って出せます。

  1. 髪ＩＫ_R/L だけ CCD フォールバックに戻す
  2. FABRIK を維持しつつ、髪チェーンにだけ安定なツイスト基準を入れる

#  FABRIK を維持しつつ、髪チェーンにだけ安定なツイスト基準を入れる手法とと、3チェーン以上は CCD を使う手法との比較

    1. FABRIK を維持しつつ、髪チェーンにだけ安定なツイスト基準を入れる

  - 利点
      - FABRIK の強みを残せます。到達性が高く、長いチェーンでも位置解がきれいに出ます。
      - 髪のような「長い・細い・ほぼ一直線」チェーンで起きやすい、ねじれ軸の反転を抑えられます。
      - 3リンクでも7リンクでも同じ解法系を使えるので、挙動の一貫性があります。

  - 欠点
      - 実装が重いです。単に FABRIK へ戻すより、ツイスト軸の保存方法を設計する必要があります。
      - 典型的には「前フレームのフレームを引き継ぐ」「parallel transport を使う」などが必要で、コードもテストも増え
        ます。

      - コリニア近傍、目標が背後に回る、左右対称モデルの鏡映など、例外条件が増えます。
      - 物理との連成で、回転は滑らかでも接触が不安定なままになる可能性はあります。根本の「物理に渡る姿勢のブレ」を止
        めるには追加検証が要ります。

  2. 3チェーン以上は CCD にする

  - 利点
      - 実装が単純です。今の source/ik.js の分岐を条件追加するだけで済みます。
      - 髪チェーンのような長い IK を、回転復元の曖昧さが少ない既存 CCD 経路に戻せます。
      - 回帰リスクが低いです。FABRIK 特有のツイスト不安定化を避けられます。
      - デバッグしやすいです。問題が出ても「長いチェーンは CCD」という明確なルールで切れます。

  - 欠点
      - FABRIK の利点を長いチェーンで捨てます。
      - 長いチェーンほど CCD は収束に反復とダンピング調整が必要で、見た目の追従が FABRIK より少し鈍くなることがあり
        ます。

      - 既定挙動が「チェーン長でアルゴリズムが変わる」ので、同じ種類のボーンでも長さでアニメーション印象が変わりま
        す。

      - 将来、髪以外の 3リンク以上チェーンにも同じ制約がかかるため、一般化すると他のモデルで思わぬ差が出ます。

  この件への適合性

  - 今回の対象は 髪ＩＫ_R/L の 7 リンクです。これは「長い・無制限・ほぼ一直線・左右対称」で、FABRIK の回転復元が不安
    定になりやすい条件です。

  - なので、短期の修正としては 3チェーン以上は CCD の方が堅いです。
  - ただし、将来的に「長いチェーンでも FABRIK を使いたい」「髪の見た目をもっと滑らかにしたい」なら、FABRIK + 安定ツ
    イスト基準 の方が本筋です。

  実務的な選び方

  - まず直したいだけなら 3チェーン以上は CCD
  - アルゴリズム品質を上げたいなら FABRIK + 安定ツイスト基準
  - なお今回のバグは、原因の中心が「回転復元時のツイスト不安定」なので、根治度は後者が上です。前者は安全な回避策で
    す。

# FABRIK の髪チェーン向け安定ツイスト復元

  ## Summary

  source/ik.js の FABRIK 経路は位置解の復元後に回転を再構成していますが、髪のような長い・ほぼ一直線・無制限の
  チェーンでは twist 軸が不定になりやすいです。
  このプランでは FABRIK を維持しつつ、髪チェーンに限って「前フレーム由来の参照軸」を使う安定ツイスト基準を導入し、
  右髪２ / 左髪２ のような剛体振動を抑えます。

  ## Key Changes

  - source/ik.js
      - FABRIK の回転再構成を、単純な currentDirection -> desiredDirection の最短回転だけに依存しない形へ拡張す
        る。

      - 各 link ごとに「前回のローカル基底」または「ボーンの初期ローカル基底」から安定参照軸を作り、twist 成分を投
        影して保持する。

      - dot ≈ -1 の反転近傍では、固定の直交軸選択ではなく、参照基準に最も近い安定軸を選ぶ。
      - 髪チェーン判定は、PMX のボーン名ベースで 髪ＩＫ 系を優先し、一般チェーンには影響を広げない。

  - source/model-scene.js
      - 既存のボーン初期基底・回転ロック情報を利用して、IK 用の安定参照基底を持てるようにする。
      - 必要なら IK chain の scene state に「安定 twist 用の参照ベクトル」か「参照 quaternion」を保持する。

  - tests/
      - テストデータの 髪ＩＫ_R/L を対象に、同一入力を複数フレーム流したときに 右髪２ / 左髪２ 付近の剛体姿勢が発散しないことを確認する回帰テストを追加する。

      - FABRIK の既存テストは維持し、reachable / unreachable の位置収束が壊れていないことも確認する。

  - docs/openmmd-specification.md
      - IK の説明を、FABRIK 位置解 + 安定 twist 復元という実装に合わせて更新する。

  ## Test Plan

  - tests/fabrik-ik-solver.test.mjs
      - 長い無制限チェーンで、同じターゲットに対する連続更新後の回転変動が小さいことを確認する。

  - tests に新規回帰テストを追加
      - テストデータを読み込み、髪ＩＫ_R/L の chain に対して IK を数フレーム反復しても、右髪２ / 左髪２ 近傍の剛体がフレームごとに大きく跳ばないことを確認する。

      - reachable / unreachable の両方で、effector が破綻せず、髪先の位置が単調に収束することを確認する。

  - 既存回帰の再実行
      - 膝や腕の IK テストが劣化していないことを確認する。
      - 物理モード関連の既存テストが通ることを確認する。

  ## Assumptions

  - 対象はまず 髪ＩＫ_R/L のような髪チェーンに限定する。
  - 髪チェーン判定はモデル名ではなくボーン名・chain 構造を使う。
  - 既存の CCD fallback は残し、FABRIK のみを改善する。
  - 3リンク以上は CCD のような大きなアルゴリズム切替は行わず、FABRIK を主経路として維持する。

  # FABRIK が暴れる原因

  • 原因は FABRIK が PMX の髪 IK の links 順を逆に解釈していたことでした。右髪ＩＫ の実データは effector 側から root
  側へ 132 -> 131 -> ... -> 126 で並んでいますが、[source/ik.js](D:/data/program/openmmd/source/ik.js) の FABRIK 経
  路は root-to-effector 前提で位置列と回転再構成をしていたので、IK ターゲットを少し動かしただけで chain 全体が不正な
  方向に解かれ、physics を切っても髪が暴れていました。

  修正では FABRIK のみ links を階層から判定して root-to-effector に正規化してから解くようにしました。あわせて、前回
  入れた髪向け stable twist はそのまま残しています。回帰として [tests/fabrik-ik-solver.test.mjs](D:/data/program/
  openmmd/tests/fabrik-ik-solver.test.mjs) に PMX 型の reverse link order を正規化するテストを追加し、既存の [docs/
  openmmd-specification.md](D:/data/program/openmmd/docs/openmmd-specification.md) の記述も維持しています。
